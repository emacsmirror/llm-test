#+TITLE: llm-test: LLM-Driven Testing for Emacs
#+AUTHOR: Andrew Hyatt

* Overview

~llm-test~ is an Emacs library that uses LLM agents to interpret natural-language
test specifications and execute them against a fresh Emacs process.  Tests are
written in YAML and registered as ERT tests, integrating with the standard Emacs
test infrastructure.

* How It Works

1. You write test specs in YAML files describing what to test in plain English.
2. ~llm-test~ parses these YAML files and registers an ERT test for each spec.
3. When a test runs, a fresh Emacs process (~emacs -Q~) is launched as a daemon.
4. An LLM agent interprets the test description, drives the fresh Emacs via
   tool calls (eval elisp, send keys, inspect buffers, etc.), and determines
   whether the test passes or fails.

** The Agent Loop

When an ERT test runs, ~llm-test~ sends the setup instructions and test
description to the LLM as a chat prompt with a set of tools.  The LLM responds
with tool calls, which ~llm-test~ executes and returns the results.  This
continues in an async loop -- each iteration is one LLM request/response round
trip -- until the agent calls ~pass-test~ or ~fail-test~, or the iteration limit
(~llm-test-max-iterations~, default 80) is reached.

** Available Tools

On each iteration the agent may call any of these tools:

- *eval-elisp* -- Evaluate arbitrary Emacs Lisp in the test subprocess and
  return the printed result.  Used for programmatic setup, state inspection, and
  anything that doesn't require simulating user input.
- *send-keys* -- Queue a key sequence (in Emacs ~kbd~ notation, e.g. ~C-x C-f~,
  ~M-x~, ~RET~) into the test Emacs as if typed by a user.  This is
  non-blocking: it schedules the keys via ~unread-command-events~ and returns
  immediately.
- *suggest-improvement* -- Record a UI/UX suggestion about the package under
  test.  This does not affect the pass/fail verdict.
- *pass-test* / *fail-test* -- Declare the test outcome and end the loop.

** How the Agent "Sees" Emacs

The test Emacs runs as a daemon with a frame sized to ~llm-test-frame-width~ x
~llm-test-frame-height~ (default 80x40).  The agent never sees a graphical
display.  Instead, every tool response automatically includes a JSON snapshot of
the entire frame state, appended after the tool's own result.  This means the
agent gets a fresh view of Emacs after every action without needing to
explicitly request it.

The frame state JSON has this structure:

#+begin_src json
{
  "windows": [
    {
      "buffer": "<buffer name>",
      "mode": "<major mode>",
      "selected": true,
      "point": 1,
      "contents": "<visible text in window>"
    }
  ],
  "minibuffer": {
    "active": false
  }
}
#+end_src

When the minibuffer is active (e.g. after ~M-x~ or a ~completing-read~ prompt),
the ~minibuffer~ object includes ~prompt~ and ~input~ fields showing the prompt
text and any input entered so far.

Each window's ~contents~ field contains only what is visible on screen (between
~window-start~ and ~window-end~), like a human looking at the Emacs frame.  The
agent can scroll with ~send-keys~ and then call any tool to get an updated
snapshot.

The agent can also use ~eval-elisp~ to query any programmatic state -- point
position, variable values, overlays, text properties, etc.

* YAML Test Format

The YAML format consists of a single group with a shared setup and a list of tests.

Each test is a natural language description of the test to run. The test
description should include the expected results.

If a test is expected to fail, the ~expected-failure~ key can be set to ~true~.
This is useful for regression testing against known issues or for testing
the test framework itself.

#+begin_src yaml
group: auto-fill mode
setup: |
  Enable auto-fill-mode in a text-mode buffer.
  Set fill-column to 40.
tests:
  - description: |
      Type a long paragraph that exceeds fill-column.
      Verify that the text is automatically wrapped.
  - description: |
      Type a numbered list item that exceeds fill-column.
      Verify that continuation lines are indented properly.
  - description: |
      Type text and verify a result that should NOT happen.
    expected-failure: true
#+end_src

* Installation

This package requires the [[https://github.com/ahyatt/llm][llm]] package and [[https://github.com/zkry/yaml.el][yaml.el]].

It is available on MELPA, so assuming you have added MELPA to your allowed archives, you can install with: 

#+begin_src emacs-lisp
(use-package llm-test)
#+end_src

The package is mainly for use by other package authors.

* Usage

#+begin_src emacs-lisp
(require 'llm-test)
(require 'llm-openai)  ; or another provider

(setq llm-test-provider (make-llm-openai :key "your-key"))
(llm-test-register-tests "path/to/testscripts/")

;; Then run with M-x ert or (ert-run-tests-batch)
#+end_src

If you don't pass ~:provider~ and don't set ~llm-test-provider~, ~llm-test~
will look for an environment variable named ~LLM_TEST_PROVIDER_ELISP~.  Its
value should be trusted elisp that constructs a provider object.

For command-line usage, this makes it easy to run ~eldev test~ or batch ERT
without editing your init files:

#+begin_src sh
export LLM_TEST_PROVIDER_ELISP='(progn
  (load "~/.emacs.d/claude-oauth.el")
  (make-llm-claude-oauth :chat-model "claude-sonnet-4-6"))'
eldev test
#+end_src

If no provider is configured by any of these methods, the registered
~llm-test~ scenarios fail with a clear error.

* Testing Your Own Package

When testing a package that isn't installed system-wide, pass ~:load-path~ to
make it available in the test Emacs subprocess, and ~:init-forms~ for any
startup code (e.g., requiring packages, configuring variables):

#+begin_src emacs-lisp
(llm-test-register-tests "path/to/testscripts/"
  :load-path '("/path/to/my-package" "/path/to/its/dependencies")
  :init-forms '((require 'my-package)
                (setq my-variable "value")))
#+end_src

The load-path entries are added and init-forms are evaluated when each test
Emacs subprocess starts, before any test setup or execution.

The provider is resolved when each test runs, in this order:
1. the explicit ~:provider~ argument,
2. ~llm-test-provider~,
3. ~LLM_TEST_PROVIDER_ELISP~.
